# Шаблон Document-Driven Development

Комплексный фреймворк для управления разработкой ПО через структурированную документацию. Этот шаблон поддерживает рабочие процессы **SDD**, **DDD**, **TDD**, **VDD**, **ADR** (Architecture Decision Records) и инструменты оркестрации (**roadmap**, **legacy**).

## Обзор

Этот шаблон рассматривает документацию как первичный артефакт разработки, где код — это реализация «последней мили», полученная из хорошо определённых документов. Он обеспечивает:

- **Возобновляемость**: Продолжение работы между сессиями без потери контекста
- **Трассируемость**: Каждое решение по реализации восходит к требованиям
- **Итеративность**: Уточнение документов перед написанием кода
- **Параллелизация**: Несколько агентов могут работать по одним документам
- **Коммуникация с заинтересованными лицами**: Презентации функциональности для клиентов и руководителей
- **История решений**: ADR фиксируют архитектурные решения с полным обоснованием

## Автоматически загружаемый контекст

Файл `CLAUDE.md` автоматически загружается в начале сессии, предоставляя:
- Сводку ADR Index (все решения со статусом и ссылками)
- Список активных потоков (SDD/DDD/TDD/VDD в процессе)
- Быструю навигацию по документации и командам

## Поддерживаемые рабочие процессы

### SDD (Spec-Driven Development)
Внутренние функции, ориентированные на разработчиков. 4 фазы:

```
REQUIREMENTS → SPECIFICATIONS → PLAN → IMPLEMENTATION
```

**Использовать когда**: Внутренняя логика, инфраструктура, инструменты разработчика.

### DDD (Document-Driven Development)
Функции, требующие **согласования с заинтересованными лицами**. 5 фаз:

```
REQUIREMENTS → SPECIFICATIONS → PLAN → IMPLEMENTATION → DOCUMENTATION
```

**Использовать когда**: Функции для клиентов, нужно «продать» функциональность, документация является deliverable.

**Ключевое отличие**: Финальная фаза создаёт **мини-презентацию** для клиентов/руководителей — не техническую документацию, а объяснение ценности.

### TDD (Tests-Driven Development)
Подход **cases-first** для функций, критичных к корректности. 6 фаз:

```
REQUIREMENTS → TESTS → SPECIFICATIONS → PLAN → IMPLEMENTATION → DOCUMENTATION
```

**Использовать когда**: Финансовые расчёты, логика безопасности, сложные state machines.

**Ключевое отличие**: Фаза TESTS — это исчерпывающий поведенческий анализ. Дизайн **возникает из cases**, не изобретается независимо. Specs **выводятся из tests**.

### VDD (Visual-Driven Development)
Функции, где первичен UI/UX. 6 фаз:

```
REQUIREMENTS → VISUAL → SPECIFICATIONS → PLAN → IMPLEMENTATION → DOCUMENTATION
```

**Использовать когда**: Пользовательские интерфейсы, визуальный дизайн является первичной заботой.

**Ключевое отличие**: Фаза VISUAL определяет UI перед техническими спецификациями.

## Быстрый старт

### 1. Выберите ваш IDE
Этот шаблон поддерживает несколько AI-powered IDE:
- **Claude Code**: Команды в `.claude/commands/`
- **Cursor**: Команды в `.cursor/prompts/commands/`
- **Qwen**: Команды в `.qwen/commands/`
- **Gemini**: Команды в `.gemini/commands/`

### 2. Запустите новый поток

```
/sdd start [feature-name]    # Внутренняя функция
/ddd start [feature-name]    # Функция для клиентов
/tdd start [feature-name]    # Критично к корректности
/vdd start [feature-name]    # Первичен UI
```

### 3. Пройдите через фазы

Каждая фаза требует явного подтверждения перед переходом:
- "requirements approved"
- "tests approved" (TDD) / "visual approved" (VDD)
- "specs approved"
- "plan approved"
- "docs approved" (DDD/TDD/VDD)

## Доступные команды

### Основные потоки

| Команда | Описание |
|---------|----------|
| `/sdd start [name]` | Запустить новый поток SDD (внутренняя функция) |
| `/ddd start [name]` | Запустить новый поток DDD (для клиентов) |
| `/tdd start [name]` | Запустить новый поток TDD (критично к корректности) |
| `/vdd start [name]` | Запустить новый поток VDD (первичен UI) |
| `/[flow] resume [name]` | Возобновить существующий поток |
| `/[flow] status` | Показать все активные потоки типа |

### Architecture Decision Records

| Команда | Описание |
|---------|----------|
| `/adr start [name]` | Запустить новый ADR в фазе DRAFT |
| `/adr resume [n]` | Возобновить существующий ADR |
| `/adr quick [name]` | Создать lightweight ADR |
| `/adr list` | Показать все ADR со статусом |
| `/adr.guide` | Справка: языковые паттерны, масштабирование, миграция |

**Типы ADR**:
- `constraining` — выбирает из вариантов, сужает scope
- `enabling` — добавляет возможности, расширяет scope

**Поток**: `DRAFT → REVIEW → APPROVED | REJECTED`

### Roadmap — Оркестрация потоков (DFS)

| Команда | Описание |
|---------|----------|
| `/roadmap` | DFS к MVP: кратчайший путь к working core |
| `/roadmap [goal]` | DFS к цели: кратчайший путь к конкретной функции |
| `/roadmap status` | Показать dependency graph и прогресс |

**Режим DFS** (Depth-First):
- Находит блокирующие зависимости рекурсивно
- Завершает каждый поток ПОЛНОСТЬЮ перед переходом к следующему
- Реализует МИНИМУМ пути для достижения цели

**Пример:**
```
/roadmap "user can login"
→ sdd-database: REQ→SPEC→PLAN→IMPLEMENT ✓
→ sdd-auth: REQ→SPEC→PLAN→IMPLEMENT ✓
→ GOAL achieved!
```

### Waterfall — Оркестрация потоков (BFS)

| Команда | Описание |
|---------|----------|
| `/waterfall` | BFS: Документировать все потоки, скомпилировать layers, реализовать |
| `/waterfall compile` | Скомпилировать layer-0/1/2 из approved plans |
| `/waterfall gaps` | Показать unresolved gaps |
| `/waterfall status` | Показать текущее состояние |

**Режим BFS** (Breadth-First):
- **Phase 1-3**: Документирует все потоки (REQ→SPEC→PLAN)
- **Phase 4**: Компилирует в architectural layers (0/1/2)
- **Phase 5**: Detect & resolve gaps в source flows
- **Phase 6-7**: Реализует bottom-up (Layer 0 → Layer 1 → Layer 2)

**Layer Architecture:**
```
Layer 2: FEATURE (UI, handlers)
    ↑ depends on
Layer 1: DOMAIN (business logic, API)
    ↑ depends on
Layer 0: SHARED (config, types, DB)
```

### Legacy — Reverse Engineering

| Команда | Описание |
|---------|----------|
| `/legacy` | Анализировать весь проект, сгенерировать документы |
| `/legacy [path]` | Анализировать конкретный path |
| `/legacy [path] "focus"` | DFS: Фокус на конкретной функциональности |

**Module-first detection** — потоки на модуль, не на тип файла.

**Авто-определение типа потока по назначению**:
- SDD ← внутренняя логика, не нужна коммуникация с заинтересованными лицами
- DDD ← требует объяснения клиенту/руководителю
- TDD ← критично к корректности, tests определяют поведение
- VDD ← UI/UX является первичной заботой

**Идемпотентность**: Добавляет к существующим потокам, без дубликатов.

Все сгенерированные потоки имеют статус DRAFT, требуют review.

## Выбор правильного потока

| Ситуация | Поток |
|----------|-------|
| Внутренний рефакторинг | SDD |
| Инструменты разработчика | SDD |
| Инфраструктура | SDD |
| Функция для клиентов | DDD |
| Функция требует «продажи» заинтересованным лицам | DDD |
| Документация является deliverable | DDD |
| Финансовые/безопасности расчёты | TDD |
| Сложные state machines | TDD |
| Поведение должно быть исчерпывающе определено | TDD |
| UI/UX является первичной заботой | VDD |
| Визуальный дизайн определяет разработку | VDD |

## Выбор оркестрации: Roadmap vs Waterfall

| Ситуация | Оркестрация |
|----------|-------------|
| Нужен быстрый результат (MVP) | `/roadmap` |
| Конкретная цель/функция | `/roadmap [goal]` |
| Минимум кода для working feature | `/roadmap` |
| Полное планирование проекта | `/waterfall` |
| Несколько независимых функций | `/waterfall` |
| Оптимизация по layers | `/waterfall` |
| Нужно избежать gaps/conflicts | `/waterfall` |

**DFS (/roadmap)**: Глубоко в один поток, завершить полностью, перейти к следующему.
**BFS (/waterfall)**: Все потоки документировать, скомпилировать в layers, реализовать bottom-up.

## Переходы между фазами

Каждый переход требует явного подтверждения пользователя:

| Переход | Фраза подтверждения |
|---------|---------------------|
| Requirements → next | "requirements approved" |
| Tests → Specs (TDD) | "tests approved" |
| Visual → Specs (VDD) | "visual approved" |
| Specs → Plan | "specs approved" |
| Plan → Implementation | "plan approved" |
| Implementation → Docs | "ready for docs" |
| Docs → Complete | "docs approved" |

## Отслеживание статуса

Каждый поток поддерживает `_status.md`:
- Текущая фаза и статус
- Чек-лист прогресса
- Blockers
- Заметки контекста для возобновления
- Следующие действия

## Лучшие практики

### Делать
- Обновлять `_status.md` после значимых изменений
- Ждать явного подтверждения перед переходом фазы
- Использовать TDD для критичной к корректности логики
- Использовать DDD когда нужно согласование с заинтересованными лицами
- Думать module-first, не file-type-first

### Не делать
- Пропускать фазы
- Создавать отдельные потоки для tests/UI модуля
- Предполагать подтверждение без подтверждения
- Использовать технический жаргон в фазе документации DDD
- Запускать /legacy без review результатов

## Передача сессии

При завершении mid-flow:
1. Обновить `_status.md` с текущим состоянием
2. Задокументировать reasoning in-flight
3. Список явных следующих шагов

Новые сессии читают `_status.md` первыми для восстановления контекста.
