# Поток Architecture Decision Records

Вы входите в режим ADR (Architecture Decision Records). Прочитайте `flows/adr.md` для полной справки по потоку.

## Команда: $ARGUMENTS

Распарсите аргументы для определения действия:

### `start [name]` — Запустить новый ADR
1. Прочитать `flows/adr-index.md` для получения следующего номера ADR
2. Создать директорию `flows/adr-[NNN]-[name]/`
3. Скопировать шаблоны из `flows/.templates/adr/`
4. Создать `_status.md` с phase = DRAFT
5. Обновить `adr-index.md` новой записью
6. Начать drafting контекста решения с пользователем

### `resume [number-or-name]` — Возобновить существующий ADR
1. Найти ADR по номеру или имени в `flows/adr-*/`
2. Прочитать `_status.md` для определения текущей фазы
3. Прочитать все существующие артефакты
4. Сообщить пользователю текущее состояние
5. Продолжить с места остановки

### `quick [name]` — Создать lightweight ADR
1. То же как `start`, но использовать шаблон `lightweight.md`
2. Упрощённо для меньших решений

### `list` — Показать все ADR
1. Прочитать `flows/adr-index.md`
2. Отобразить таблицу: Number | Title | Status | Created

### `status` — Показать активные ADR
1. Список ADR в фазе DRAFT или REVIEW
2. Показать текущие blockers и следующие действия

### Нет аргументов или `help`
1. Показать доступные команды
2. Показать сводку ADR по статусу (draft/review/approved/rejected)

---

## Поведения фаз

### Фаза DRAFT
- Получить контекст и decision drivers от пользователя
- Задокументировать рассмотренные варианты (минимум 2)
- Проанализировать pros/cons для каждого варианта
- Draft предложенного решения
- Задокументировать ожидаемые последствия
- Итеративно обновлять `adr.md`
- Ждать явного "ready for review" перед переходом

### Фаза REVIEW
- Проверить все требуемые секции завершёнными
- Задать уточняющие вопросы если нужно
- Address feedback от reviewer
- Обновить consequences на основе review
- Ждать явного "ADR approved" или "ADR rejected"

### Фаза APPROVED
- Решение окончательно
- Обновить `adr-index.md` с decided date
- Ссылка на implementing specs/PRs когда доступны
- Пометить любые superseded ADR

### Фаза REJECTED
- Задокументировать rationale отклонения
- Заметить альтернативное направление если применимо
- Сохранить для исторической справки

---

## Переходы статуса

```
DRAFT ──["ready for review"]──> REVIEW
REVIEW ──["ADR approved"]──> APPROVED
REVIEW ──["ADR rejected"]──> REJECTED
APPROVED ──[новый ADR supersedes]──> SUPERSEDED
```

---

## Всегда

- Обновлять `_status.md` после каждого значимого изменения
- Обновлять `adr-index.md` при изменении статуса
- Никогда не пропускать фазы и не предполагать подтверждение
- При неуверенности спрашивать, а не предполагать
- Ссылаться на связанные ADR и specs
- Держать решение сфокусированным (одно решение на ADR)
